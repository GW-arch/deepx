5월 8일 작업은 `air_drum_pad`를 단순 실시간 데모에서 **측정 가능한 CPU/NPU 하이브리드 손 추적 파이프라인**으로 끌어올린 작업이다. 핵심은 offline dataset benchmark, palm 재검출/비동기 실험, NPU landmark 보정, 단위 테스트와 품질 체크 스크립트까지 이어지는 반복 검증 루프를 만든 점이다.

## 실행 가능 상태

현재 레포는 같은 `main.py`에서 air drum과 piano를 모두 실행할 수 있다.

| 모드 | 실행 방식 | 비고 |
|------|-----------|------|
| Air drum | `python3 main.py --backend cpu --camera 0` | `--piano`를 붙이지 않으면 기본 드럼 매핑 |
| Piano | `python3 main.py --backend cpu --piano --camera 0` | 기본 10키 매핑(C4-E5) 사용 |
| NPU piano | `./scripts/run_npu_full_piano.sh` | Palm+Hand 정식 파이프라인, correction 자동 적용 가능 |
| NPU drum | `python3 main.py --backend npu-full ...` | 피아노와 같은 백엔드에서 `--piano` 없이 실행 |

NPU 계열은 `.dxnn`, TFLite 모델, DX-RT 환경이 필요하다. 실제 연주 데모는 카메라, 디스플레이, 오디오 장치가 준비되어야 한다.

## 기능 단위 분석

### 1. Offline benchmark 루프

`tools/benchmark_dataset.py`가 추가되면서 라이브 카메라 없이 `dataset/frame_*.png`를 재생해 백엔드별 지연과 landmark 차이를 비교할 수 있게 되었다.

지원된 주요 옵션:

- `--backends cpu-baseline,npu-full`
- `--palm-redetect-every N`
- `--async-palm`
- `--frame-interval-ms`
- `--csv`, `--json`
- `--debug-dir`, `--debug-top-k`
- `--landmark-correction`

이 변화의 의미는 크다. 이전에는 "실행해 보니 빠르다/느리다" 수준의 관찰에 가까웠다면, 이제 같은 입력 프레임에서 CPU 기준선과 NPU 결과를 정량 비교할 수 있다.

### 2. Palm 병목과 ROI tracking 실험

`npu-full`과 `cpu-baseline`은 모두 Palm Detection -> ROI crop -> Hand Landmark 구조를 공유한다. 5월 8일 작업에서 palm detection을 매 프레임 실행하는 기본 경로와, `--palm-redetect-every N`으로 palm을 건너뛰는 실험 경로가 분리됐다.

현재 기본값은 정확도 우선이다.

- `--palm-redetect-every 0`: 매 프레임 palm detection
- `N > 0`: 이전 landmark에서 ROI를 추적하고 일정 프레임 동안 palm detection 생략

Palm skip은 평균 지연을 크게 줄일 수 있지만, NPU INT8 landmark 편향이 누적되면 ROI drift가 생긴다. 그래서 기본값으로 채택하지 않고 실험 옵션으로 남긴 판단이 적절하다.

### 3. Async palm 실험

`FullNpuHandsTracker`에 background palm 실행 경로가 추가됐다. `--async-palm`을 켜면 메인 루프는 이전 ROI로 hand landmark를 계속 추적하고, palm detection은 별도 thread에서 새 ROI를 준비한다.

의도는 palm detection의 긴 지연을 프레임 루프 밖으로 밀어내는 것이다. 다만 정확도와 안정성은 입력 pacing, 손 이동 속도, ROI drift에 민감하므로 현재는 실험 기능으로 보는 것이 맞다.

### 4. NPU landmark correction

`tools/calibrate_npu_landmarks.py`와 `models/npu_landmark_correction.dataset.json`이 추가됐다. CPU TFLite baseline을 기준으로 NPU landmark의 xy 편향을 affine correction으로 보정한다.

현재 90프레임 training set 기준 결과:

| 손 | 보정 전 평균 오차 | 보정 후 평균 오차 | 보정 전 손끝 오차 | 보정 후 손끝 오차 |
|----|------------------:|------------------:|------------------:|------------------:|
| Right | 0.0270 | 0.0102 | 0.0336 | 0.0112 |
| Left | 0.0256 | 0.0092 | 0.0353 | 0.0090 |

평균 오차와 손끝 오차는 크게 줄었다. 하지만 같은 dataset으로 fit/eval한 결과라서 일반화 성능은 아직 검증되지 않았다. 조명, 손 크기, 카메라 거리, 손 방향이 다른 hold-out capture set이 필요하다.

### 5. 품질 체크와 테스트

`tests/`와 `scripts/check_quality.sh`가 추가됐다.

테스트 범위:

- strike detector
- palm ROI transform
- benchmark helper
- capture dataset indexing

`check_quality.sh`는 다음을 묶는다.

모델/데이터셋이 없는 환경에서는 `RUN_BENCH_SMOKE=0 ./scripts/check_quality.sh`로 smoke benchmark를 건너뛸 수 있다.

## 아키텍처 영향

5월 8일 이후 `air_drum_pad`의 백엔드 구분은 다음처럼 정리된다.

| 백엔드 | Palm Detection | Hand Landmark | 역할 |
|--------|----------------|---------------|------|
| `cpu` | MediaPipe 내장 | MediaPipe 내장 | 가장 단순한 실행/데모 경로 |
| `cpu-baseline` | CPU TFLite float32 | CPU TFLite float32 | `npu-full`과 동일 구조의 비교 기준선 |
| `npu-full` | CPU TFLite float32 | NPU `.dxnn` INT8 | 정식 Palm+Hand 하이브리드 파이프라인 |
| `npu` | 없음 | NPU `.dxnn` INT8 | palm 없는 dual-halves 근사 경로 |

중요한 판단:

- Palm `.dxnn`은 자동 선택하지 않고, 재컴파일 검증 후보를 `--palm-dxnn`으로 명시 선택한다.
- `npu-full` 자동 탐색은 Palm TFLite를 우선한다.
- NPU correction은 `npu-full`에만 적용한다.
- `cpu-baseline`은 보정 없이 비교 기준선으로 유지한다.

## 측정 결과 요약

`dataset/` 90프레임, 기본 palm every frame 기준:

| 백엔드 | 평균 | P95 | 프로파일 |
|--------|-----:|----:|----------|
| `cpu-baseline` | 84.40 ms | 86.78 ms | palm 39.26 ms + hand 44.80 ms |
| `npu-full` | 50.32 ms | 54.52 ms | palm 40.93 ms + hand 9.13 ms |

`npu-full --palm-redetect-every 5`, 20프레임 smoke:

- 평균 15.29 ms
- P95 50.96 ms
- palm frames 3/20
- tracking frames 17/20

해석:

- Hand landmark 단계는 NPU로 가속 효과가 확인된다.
- 전체 지연은 여전히 Palm Detection 비중이 크다.
- Palm skip/async palm은 지연 목표에 가까워지는 방향이지만 정확도 검증이 선행되어야 한다.

## 리스크와 남은 과제

1. Correction overfit 위험
   - 현재 보정 JSON은 같은 90프레임 dataset에 대해 학습과 평가를 같이 수행했다.
   - 별도 capture set으로 hold-out 검증이 필요하다.

2. Palm skip/async 안정성
   - 지연은 줄지만 ROI drift와 손 빠짐 가능성이 있다.
   - sweep 결과를 기준으로 기본값을 바꿀지 결정해야 한다.

3. Palm Detection 병목
   - 당시 기본 `npu-full`에서는 palm이 CPU TFLite였고 전체 latency의 주요 병목이었다.
   - 2026-06-16 재평가에서는 재컴파일한 palm `.dxnn` 후보가 offline replay를 통과했으므로, 남은 과제는 live camera 안정성과 guided hit accuracy 검증이다.

4. Live E2E 검증 부족
   - offline benchmark는 vision loop 비교에는 충분하지만, 카메라 입력부터 오디오 출력까지의 실제 E2E latency를 대체하지 않는다.
   - 고속 카메라 또는 오디오 파형 기반 측정이 필요하다.

5. 실행 스크립트 편차
   - piano용 스크립트는 `run_cpu_piano.sh`, `run_npu_piano.sh`, `run_npu_full_piano.sh`가 있다.
   - air drum은 `main.py`를 `--piano` 없이 직접 실행하면 되지만, 정식 NPU drum 실행 스크립트는 별도로 없다.

## 권장 다음 작업

1. Hold-out dataset 추가
   - 조명, 거리, 손 방향, 손 크기가 다른 capture set을 추가한다.
   - correction 적용 전/후를 training set과 hold-out set으로 분리해 비교한다.

2. Palm skip sweep 결과 문서화
   - `--palm-redetect-every 0,1,2,3,5,10` 결과를 CSV/JSON으로 저장한다.
   - 평균/P95 지연과 landmark 오차를 같이 보고 기본값 변경 여부를 판단한다.

3. Async palm live 테스트
   - `--frame-interval-ms 16.7` offline pacing과 실제 카메라 입력을 모두 비교한다.
   - 손 빠짐, ROI drift, ghost hit 여부를 기록한다.

4. Air drum NPU 실행 스크립트 추가
   - 현재 구조상 실행은 가능하지만 piano 스크립트처럼 명확한 entrypoint가 있으면 데모 운영이 쉬워진다.

5. E2E latency 측정
   - vision inference가 아니라 실제 타격 동작부터 오디오 출력까지 측정한다.
   - 프로젝트 목표인 15-20 ms 이하에 대해 현재 병목을 다시 평가한다.
