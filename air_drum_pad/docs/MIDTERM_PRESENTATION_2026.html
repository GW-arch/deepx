Presentation Outline

발표에서 답해야 하는 질문

1. 무엇을 만들었는가?

손가락 landmark를 이용해 실제 타격 동작을 인식하고, 드럼/피아노 사운드를 재생하는 가상 악기 인터페이스.

2. NPU는 어디에 쓰는가?

현재 안정 경로에서는 hand landmark CNN을 NPU `.dxnn`으로 실행하고, palm detection은 CPU TFLite로 유지한다.

3. 왜 그렇게 분할했는가?

Palm detector는 INT8 양자화 후 score head가 깨졌다. 반면 hand landmark는 NPU 가속 효과가 확인됐다.

4. 어느 정도 검증했는가?

라이브 데모 경로, dataset replay benchmark, landmark error 비교, 단위 테스트, 보정 실험을 갖췄다.

Why Palm Stays on CPU

Palm detection NPU화 실패 원인

속도 관점

  • Palm `.dxnn` 추론은 약 12 ms로 CPU TFLite보다 빠름
  • 속도만 보면 NPU로 옮기는 것이 맞아 보임
  • 하지만 detector는 score가 깨지면 box가 있어도 사용할 수 없음
Palm NPU ~12 ms
Palm CPU ~40-95 ms

정확도 관점

  • INT8 양자화 후 score head가 손 검출 점수를 잃음
  • ONNX와 NPU score correlation: -0.11
  • ema/minmax, opt level 변경으로 해결되지 않음
score head: broken
box head: partially usable

결론: 중간발표 기준 목표 아키텍처는 Palm까지 전부 NPU가 아니라, Palm은 CPU 안정 경로, Hand landmark는 NPU 가속이다.

Latency Optimization

Palm 병목을 줄이는 실험 구조

Palm Redetect Interval

매 프레임 palm detection을 하지 않고, 일정 프레임 동안 이전 landmark에서 ROI를 재추정한다.

  • 장점: 평균 latency 감소
  • 위험: NPU landmark 편향이 누적되어 ROI drift 발생 가능
  • 검증: `0,1,2,3,5,10` sweep으로 latency/error curve 작성

Async Palm

Palm detection은 background thread에서 돌리고, foreground는 이전 ROI로 hand landmark를 계속 추적한다.

  • 장점: frame loop의 체감 latency를 줄일 수 있음
  • 위험: 빠른 손 움직임에서 stale ROI 사용
  • 검증: camera pacing과 실제 live input 모두 비교 필요

두 방법 모두 “기본 아키텍처”가 아니라 최적화 실험이다. 교수/리뷰어에게는 기본값이 정확도 우선임을 먼저 밝히는 것이 안전하다.

Quality and readiness
Architecture Risks

리뷰어가 물을 가능성이 높은 지점

1. Palm이 CPU면 NPU 프로젝트인가?

Hand landmark CNN은 NPU로 실행한다. Palm까지 NPU화는 시도했으나 detector score 품질 문제로 안정 경로에서 제외했다.

2. 보정 결과가 overfit 아닌가?

맞다. 현재는 같은 dataset에서 가능성 확인 단계이며, hold-out dataset이 다음 필수 작업이다.

3. offline benchmark가 실제 연주 지연을 대체하나?

대체하지 않는다. vision pipeline 비교용이고, 최종 평가는 camera-to-audio E2E 측정이 필요하다.

4. 최종 데모 전략은?

CPU path로 안정 데모를 확보하고, NPU path는 runtime 복구 후 수치 비교와 함께 보여주는 전략이 안전하다.
